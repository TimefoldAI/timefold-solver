[#overconstrainedPlanning]
= Overconstrained planning
:doctype: book
:sectnums:
:icons: font

Sometimes there are not enough resources to assign all planning entities.
For example, there may be more shifts than available employees, or more deliveries than vehicles can handle in a day.
This is called an overconstrained problem.

By default, Timefold Solver assigns every planning entity to a value, which means it will overload resources and break hard constraints rather than leave anything unassigned.
The correct approach is to explicitly model the possibility of leaving entities unassigned.

[#overconstrainedPlanningWithNullValues]
== The recommended approach

There are two steps to handle overconstrained planning correctly:

. Allow the planning variable to be unassigned.
. Add a score constraint to penalize unassigned entities.

=== Step 1: Allow unassigned values

Set `allowsUnassigned = true` on the planning variable:

[tabs]
====
Java::
+
[source,java,options="nowrap"]
----
@PlanningEntity
public class Shift {

    @PlanningVariable(allowsUnassigned = true)
    private Employee employee;

    ...
}
----
====

Timefold Solver automatically adds `null` to the value range.
There is no need to add `null` to any `ValueRangeProvider`.

=== Step 2: Penalize unassigned entities

Add a score constraint at the appropriate level (usually medium, between hard and soft) to penalize each unassigned entity.
Use xref:constraints-and-score/score-calculation.adoc#constraintStreamsForEach[`forEachIncludingUnassigned()`] to include entities with a `null` planning variable, which are otherwise filtered out by default:

[tabs]
====
Java::
+
[source,java,options="nowrap"]
----
private Constraint penalizeUnassignedShifts(ConstraintFactory factory) {
    return factory.forEachIncludingUnassigned(Shift.class)
            .filter(shift -> shift.getEmployee() == null)
            .penalize(HardMediumSoftScore.ONE_MEDIUM)
            .asConstraint("Unassigned shift");
}
----
====

[IMPORTANT]
====
Without this constraint, a solution with *all* entities unassigned scores better than one with any hard constraint violations — because no assignments means no hard constraint violations.
Always penalize unassigned entities when using `allowsUnassigned = true`.
====

The solver will now only leave a shift unassigned when every possible assignment would cost more in hard or medium score than leaving it empty.

[#dummyValuesAntipattern]
== The dummy values antipattern

A common alternative is to introduce a "dummy" resource (e.g. a dummy employee or a dummy vehicle) to absorb the unassigned entities.
The idea is that entities assigned to the dummy are effectively unassigned.

This approach appears to work at first, but causes a range of problems.

=== Hard constraints bleed onto the dummy

Real constraints (e.g. skill requirements, capacity limits, or working hour rules) now also apply to the dummy resource.
Either the dummy violates them (corrupting the score), or you add `!= dummyEmployee` guards to every constraint that touches that relationship.
As the constraint set grows, these guards are easy to forget and hard to test.

=== The search space grows unnecessarily

The solver treats the dummy resource as a valid assignment target like any other value.
Entities are considered for assignment to the dummy even when a real assignment is available, which inflates the search space and slows down solving.

=== Solutions are harder to reason about

From the solver's perspective, an entity assigned to the dummy is fully assigned.
It takes extra application logic to distinguish "assigned to dummy" from "genuinely assigned", both in constraints and in the output of the solver.

=== Use `allowsUnassigned` instead

The `allowsUnassigned = true` approach avoids all of these problems:

* `null` is treated as a first-class outcome, not a workaround.
* Constraints do not need to be aware of a dummy value.
* The search space is not inflated.
* Unassigned entities are unambiguous in the solution output.
